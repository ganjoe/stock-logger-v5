1. Einführung

stock-logger-v5 ist eine umfassende Aktienhandels- und Überwachungsanwendung, die folgende Hauptkomponenten vereint:

  • Broker-Integration: Direkte Anbindung an Interactive Brokers (IBKR) über das IBKR Gateway.
  • Handelsverwaltung: Erstellung, Überwachung und Schließung von Positionen mit verschiedenen Order-Typen (Market, Limit, Stop, Stop-Limit).
  • Risikomanagement: Berechnung von Positionsgrößen nach Minervini-Prinzipien, Heat-Index, R-Multiple und mehr.
  • Portfolio-Analytik: Live- und historische Portfolio-Snapshots mit detaillierten Metriken.
  • Personal Trading Assistant (PTA): Ein KI-gestützter Assistent (LLM), der über natürliche Sprache mit dem System interagiert.

Die Anwendung unterstützt sowohl einen Bot-Modus (BOT) mit reinem JSON-Output für den Headless-Betrieb als auch einen Human-Modus (HUMAN) mit formatierter Ausgabe.

Bot-First Principle und Headless-Betrieb

Das System ist für den Betrieb auf Headless Linux Servern (ohne GUI) konzipiert. Das Bot-First Principle bedeutet:

  • PTA als einziges Frontend: Alle Interaktionen erfolgen natürlichsprachlich über den Personal Trading Assistant.
  • Kein String-Parsing: Bots parsen niemals Text. Das Frontend (PTA) konsumiert ausschließlich JSON.
  • SSH-Interaktion: Die primäre Schnittstelle ist eine SSH-Terminal-Session.
  • Terminal 1 (Control): Interaktiver Chat mit dem PTA (chat).
  • Terminal 2 (Debug): Live-Monitoring der Kommunikation (tail -f logs/communication.log).

Die Kommunikation zwischen PTA und Backend (Prompt → JSON-Command → JSON-Response) wird transparent geloggt und ist in Echtzeit nachvollziehbar.

3. ALM-Prinzipien

Das System folgt den Architektur- und Design-Prinzipien, die in alm_prinzipien.md dokumentiert sind.

3.1 Verbindungsmanagement (Connection 3.0 - PTA Driven)

Die technische Verbindung wird nicht mehr durch das Start-Skript erzwungen. Stattdessen startet das System "Offline" und der PTA übernimmt die Verantwortung für den Verbindungsaufbau:

  1. Systemstart (run_logger.py oder run_pta.py)
  2. PTA führt Auto-Discovery durch:
    • Versuch 1: Connect localhost:4002 (Paper Default)
    • Versuch 2: Connect localhost:4001 (Live Fallback)
    • Versuch 3: Nachfrage beim User (Custom IP/Port)
  3. Erfolg/Misserfolg wird im Chat gemeldet

Single Writer (ID 0)

Schreibzugriffe (Orders) nutzen strikt Client ID 0.

Multi-Reader (Bulk Fetch)

Für Massendatenabrufe (bulk_fetch) werden temporäre Sessions mit separaten IDs genutzt, um Pacing-Limits zu umgehen.

3.2 Bot-First Principle (Headless & ChatOps)

Siehe Abschnitt 1. Einführung für detaillierte Erklärung des Bot-First Principle und Headless-Betriebs.

3.3 Historische Integrität (Point-in-Time Analysis)

  • Log-basierte Rekonstruktion: Der Zustand des Portfolios zu einem beliebigen Zeitpunkt T wird durch Replay aller Transaktionen ermittelt.
  • Immutability: Einmal geschriebene Order-Logs werden niemals gelöscht oder verändert. Korrekturen erfolgen durch neue, kompensierende Einträge.
  • Zeitstempel-Filterung: Jede Analyse-Funktion muss einen date-Parameter akzeptieren.
  • LIFO-Konsistenz: Die Methode der Bestandsbewertung (LIFO) ist systemweit konsistent.

3.4 Resilienz & Daten-Sicherheit

  • Unabhängigkeit von Live-Daten: Historische Analysen funktionieren auch ohne Broker-Verbindung.
  • Graceful Adapter Failure: Broker-Adapter bringen das System nicht zum Absturz bei fehlenden Daten.
  • Atomares Speichern: Trade-Daten werden atomar via Temp-Dateien und os.rename gespeichert.
  • UUID-Garantie: Jedes Datenobjekt erhält bei der Erstellung eine unveränderliche UUID.

4. Installation und Einrichtung

Voraussetzungen

  • Python 3.9+
  • Interactive Brokers Gateway oder TWS installiert
  • Docker und Docker Compose (für IB-Gateway Container)
  • LM Studio (für den PTA/KI-Assistenten)

Installationsschritte

  1. Abhängigkeiten installieren: pip install -r requirements.txt
  2. IB Gateway Container starten: docker-compose up -d
  3. LLM-Konfiguration erstellen: py_pta/llm_config.json oder secrets/llm_config.json
  4. Hauptskript ausführen: python run_logger.py, python main_cli.py oder python run_pta.py

Docker-Konfiguration

Die docker-compose.yml definiert zwei IB-Gateway-Container:

  • ib-gateway_live: Für das Live-Trading (Port 4001)
  • ib-gateway_paper: Für das Paper-Trading/Testing (Port 4002)

5. Start-Optionen

run_logger.py

Das klassische Startskript für den Stock-Logger. Unterstützt --mode=human und --mode=bot.

Beispiel: python run_logger.py --mode=human

run_pta.py

Das eigenständige PTA-Startskript für den Telnet-basierten Betrieb. Verbindet das Trading-System mit einem TelChat Hub.

Syntax: python run_pta.py [--host HOST] [--port PORT] [--alias ALIAS]

Parameter:

  • --host: TelChat Hub Host (Standard: 127.0.0.1)
  • --port: TelChat Hub Port (Standard: 9999)
  • --alias: Alias beim Hub registrieren (Standard: pta)

Beispiel: python run_pta.py --host 192.168.1.100 --port 9999 --alias mypta

Die Standard-Konfiguration wird aus py_pta/telnet_config.json geladen.

pta_console.sh

Shell-Skript für die PTA-Konsole, ideal für SSH-Sessions. Startet den Stock-Logger mit Client ID 99, um Konflikte zu vermeiden.

Verwendung: ./pta_console.sh

Erkennt automatisch die virtuelle Umgebung (.venv) und startet mit run_logger.py --mode=human.

6. Hauptfunktionen

6.1 Handelsverwaltung

Das System verwaltet den vollständigen Lebenszyklus einer Handelstransaktion:

  1. PLANNED: Watchlist/Beobachtete Position, noch keine Order gesendet
  2. OPENING: Entry-Order gesendet, noch keine Ausführung
  3. OPEN: Position aktiv (teilweise oder vollständig gefüllt)
  4. CLOSING: Exit-Order gesendet
  5. CLOSED: Position geschlossen, Handel abgeschlossen
  6. ARCHIVED: Historischer Eintrag

Unterstützte Order-Typen:

  • Market Order: Sofortige Ausführung zum aktuellen Marktpreis
  • Limit Order: Ausführung nur zum angegebenen Preis oder besser
  • Stop Order: Wird aktiv, wenn der Stop-Preis erreicht ist
  • Stop-Limit Order: Kombination aus Stop und Limit

6.2 Portfolio-Überwachung

Der Live-Portfolio-Manager (LivePortfolioManager) erfasst folgende Daten:

  • Gesamter Kontostand (Net Liquidation, Total Cash)
  • Aktive Positionen mit Durchschnittspreis und aktuellen Marktpreis
  • Offene Orders mit Details (Order-ID, Typ, Preis)

Der Snapshot wird automatisch in data/portfolio_latest.json gespeichert.

6.3 Analytik und Risikomanagement

Das Risikomanagement-Modul bietet:

  • Heat-Index: Gesamtes Risiko aller Positionen als Prozentsatz des Kontowerts
  • R-Multiple: Verhältnis von Gewinn/Verlust zum initialen Risiko
  • Minervini Sizing: Positionsgrößen-Berechnung nach den Regeln von Mark Minervini
  • Winrate, Profit Factor, Expectancy, SQN: Statistische Metriken für Trade-Performance

6.4 Historische Daten

Marktdaten werden über den DataNodeProvider aus dem externen stock-data-node Parquet-Datenbank geladen. Unterstützte Timeframes:

  • 1D: Tägliche Daten (OHLCV)
  • 1H: Stündliche Daten
  • 5m: 5-Minuten-Daten

6.5 Personal Trading Assistant (PTA)

Der PTA ist ein KI-gestützter Assistent, der:

  • Natürliche Sprache versteht und CLI-Befehle daraus generiert
  • Selbstständig Handelsoperationen ausführt
  • Kontextuelles Gedächtnis über die Konversation hinweg behält
  • Sich über einen Telnet-Hub mit anderen Diensten kommunizieren kann

Der PTA verwendet LM Studio für die lokale LLM-Ausführung und unterstützt Tool-Aufrufe für CLI-Befehle.

7. Befehlsreferenz

7.1 Verbindung

connect

Stellt eine Verbindung zum IBKR Gateway her.

Syntax: connect [host|default] [port] [client_id]

Beispiele:

  • connect default - Verwendet die Standard-Konfiguration (Docker-Gateway)
  • connect 192.168.1.100 4001 22 - Verbindet mit spezifischem Host

disconnect

Trennt die aktive Verbindung zum IBKR Gateway.

Syntax: disconnect

7.2 Handel (trade)

Der trade-Befehl ist der zentrale Befehl für alle Handelsoperationen. Er erwartet ein JSON-Payload.

Aktionen

Aktion Zweck Erforderliche Felder
ENTER Position eröffnen ticker, quantity, optional: limit_price, stop_loss, trade_id
UPDATE Bestehende Position aktualisieren ticker, trade_id, optional: stop_loss
EXIT Position schließen ticker, trade_id
CANCEL Offene Order stornieren ticker, trade_id, broker_order_id
REFRESH Trade-Daten aktualisieren ticker, trade_id
CASH Ein-/Auszahlung (kein Broker) quantity (positiv=Einzahlung, negativ=Auszahlung)

Beispiele

Market Order (Kauf):

trade {"action": "ENTER", "ticker": "NVDA", "quantity": 10}

Limit Order (Kauf):

trade {"action": "ENTER", "ticker": "NVDA", "quantity": 10, "limit_price": 850.00, "stop_loss": 830.00}

Position schließen:

trade {"action": "EXIT", "ticker": "NVDA", "trade_id": "fc3f41e5-..."}

Stop-Loss aktualisieren:

trade {"action": "UPDATE", "ticker": "NVDA", "trade_id": "fc3f41e5-...", "stop_loss": 825.00}

Einzahlung:

trade {"action": "CASH", "quantity": 5000, "note": "Monatliche Einzahlung"}

7.3 Status-Befehle

status

Zeigt einen Live-Snapshot des Portfolios.

Syntax: status [ticker_filter]

Beispiele:

  • status - Gesamtes Portfolio
  • status {"ticker": "NVDA"} - Nur NVDA-Position
  • status NVDA - Kurzform für einen Ticker

trades

Listet alle offenen Positionen auf.

Syntax: trades

quote

Holt den aktuellen Marktpreis für ein Symbol.

Syntax: quote SYMBOL

Beispiel: quote NVDA

chart

Holt historische Chart-Daten für ein Symbol.

Syntax: chart SYMBOL [JSON_PAYLOAD]

Beispiel: chart NVDA {"timeframe": "1D", "lookback": "1Y"}

7.4 Analytik-Befehle

analyze

Führt Analytik auf Live- oder historischen Daten durch.

Syntax: analyze [live|history] [JSON_PAYLOAD]

Beispiele:

  • analyze live {"ticker": "AAPL"} - Risiko-Bericht für aktuelles Portfolio
  • analyze history {"days": 30} - Performance der letzten 30 Tage
  • analyze history {"days": 90, "ticker": "NVDA"} - Ticker-spezifische Historie

wizard

Berechnet die Positionsgröße nach Minervini-Prinzipien.

Syntax: wizard JSON_PAYLOAD

Beispiel:

wizard {"symbol": "NVDA", "entry": 850, "stop": 830, "risk_pct": 1.0, "max_pos_pct": 25.0, "equity_override": 100000}

Wenn entry fehlt oder 0 ist, wird der Live-Preis automatisch geholt.

market_clock

Zeigt den aktuellen Börsenstatus und Zeit bis zum nächsten Ereignis.

Syntax: market_clock

Mögliche Status: OPEN, PRE-MARKET, POST-MARKET/CLOSED, CLOSED (HOLIDAY), CLOSED (WEEKEND)

bulk_fetch

Startet einen Massen-Download von Marktdaten im Hintergrund.

Syntax: bulk_fetch [client_id]

7.5 Historie (history)

Holt einen historischen Portfolio-Snapshot zu einem bestimmten Datum.

Syntax: history [Tage_zurueck] (Standard: 0 = heute)

Beispiel: history 30 - Portfolio-Status vor 30 Tagen

7.6 PTA-Befehle

pta

Kommunikation mit dem Personal Trading Assistant.

Syntax: pta

Beispiele:

  • pta Wie sieht mein Portfolio aus?
  • pta Schließe Position NVDA mit ID fc3f41e5
  • pta Risiko meines Portfolios anzeigen

chat

Startet einen interaktiven Chat-Modus mit dem PTA.

Syntax: chat

Beenden mit exit oder quit.

8. Datenstruktur

8.1 Trade-JSON-Dateien

Jede Position wird als JSON-Datei gespeichert unter data/trades//.json.

Struktur einer Trade-JSON-Datei

{
  "id": "fc3f41e5-...",          // UUID der Position
  "ticker": "NVDA",              // Aktiensymbol
  "status": "OPEN",              // PLANNED, OPENING, OPEN, CLOSING, CLOSED, ARCHIVED
  "trade_type": "STOCK",         // STOCK oder CASH
  "transactions": [              // Alle Ausführungen
    {
      "id": "exec_id",
      "timestamp": "2024-01-15T10:30:00",
      "type": "ENTRY",           // ENTRY, EXIT, ADJUSTMENT
      "quantity": 10.0,          // Positiv = Kauf/Long-Buy, Negativ = Verkauf/Long-Sell
      "price": 850.00,
      "commission": 0.50,
      "slippage": 0.0,
      "order_id": "12345"
    }
  ],
  "order_history": [             // Alle Order-Einreichungen
    {
      "timestamp": "2024-01-15T10:29:00",
      "order_id": "12345",
      "action": "BUY",
      "status": "FILLED",
      "message": "Order ausgeführt",
      "quantity": 10.0,
      "type": "LMT",             // LMT, MKT, STP, STP LMT
      "limit_price": 850.00,
      "stop_price": null,
      "trigger_price": null,
      "note": "Initial Entry"
    }
  ],
  "active_orders": {},           // {broker_order_id: "ENTRY"|"STOP"|"EXIT"}
  "initial_stop_price": 830.00,  // Initiales Stop-Loss
  "current_stop_price": 835.00,  // Aktuelles Stop-Loss (trailing)
  "entry_date": "2024-01-15T10:30:00",
  "notes": "Gewinnwachstum im AI-Sektor",
  "currency": "USD",
  "exchange": "SMART"
}

Struktur

{

      "current_price": 870.00,
      "market_value": 8700.00,
      "unrealized_pnl": 200.00,
      "trade_id": "fc3f41e5-..."
    }
  ],
  "active_orders": [             // Offene Orders
    {
      "ticker": "AAPL",
      "order_id": "67890",
      "action": "BUY",
      "type": "LMT",
      "qty": 5.0,
      "price": 185.00,
      "trade_id": ""
    }
  ],
  "source": "LIVE"               // LIVE oder HISTORY
}

8.3 Log-Dateien

PTA-Kommunikation wird in logs/communication.log protokolliert.

Log-Format

2024-01-15 10:30:00 - [USER] Wie sieht mein Portfolio aus?
2024-01-15 10:30:01 - [PTA_THOUGHT] Calling Tool: execute_cli_command with args: {"command": "status"}
2024-01-15 10:30:01 - [SYSTEM_EXEC] status
2024-01-15 10:30:02 - [SYSTEM_RESULT] {"success": true, "payload": {...}}
2024-01-15 10:30:02 - [PTA_RESPONSE] Ihr Portfolio hat einen Gesamtwert von $125,000...

Log-Aktoren

  • [USER] - Benutzer-Eingabe an den PTA
  • [PTA_THOUGHT] - PTA plant Tool-Aufruf
  • [SYSTEM_EXEC] - Ausgeführter CLI-Befehl
  • [SYSTEM_RESULT] - Ergebnis des CLI-Befehls
  • [PTA_RESPONSE] - Finale Antwort an den Benutzer
  • [ERROR] - Fehlerereignisse

12. Konfiguration

12.1 Docker-Umgebung

Die docker-compose.yml definiert die IB-Gateway-Container:

services:
  ib-gateway_live:
    image: ...
    environment:
      - IB_GATEWAY_HOST=ib-gateway_live-ib-gateway-1
      - IB_GATEWAY_PORT=4001

  ib-gateway_paper:
    image: ...
    environment:
      - IB_GATEWAY_PORT=4002

Die Umgebungsvariablen IB_GATEWAY_HOST und IB_GATEWAY_PORT können über .env oder System-Environment überschrieben werden.

12.2 Dockerfile

Das Dockerfile (neben docker-compose.yml) definiert die Container-Build-Pipeline:

  • Base Image: python:3.11-slim
  • System-Abhängigkeiten: git
  • Python-Abhängigkeiten: requirements.txt
  • Default CMD: python run_pta.py (PTA Telnet Agent)

12.3 LLM-Konfiguration

Der PTA benötigt eine Konfigurationsdatei an einem der folgenden Orte:

  • py_pta/llm_config.json (Standard, relativ zu bridge.py)
  • secrets/llm_config.json (alternativ)
{
  "llm_base_url": "http://localhost:1234/v1",
  "llm_model_name": "your-model-name",
  "llm_gpu_offload": "max",
  "system_prompt": "...",
  "tools_schema": [...]
}

Felder:

  • llm_base_url: Die OpenAI-kompatible API-URL (LM Studio standardmäßig)
  • llm_model_name: Der Name des geladenen Modells in LM Studio
  • llm_gpu_offload: GPU-Offloading-Setting ("max" oder "min")
  • system_prompt: Die System-Anweisung für den PTA

12.4 Telnet-Konfiguration

Die Telnet-Hub-Konfiguration befindet sich in py_pta/telnet_config.json:

{
  "host": "127.0.0.1",
  "port": 9999,
  "alias": "pta"
}

Diese Konfiguration wird automatisch von run_pta.py geladen und kann über CLI-Parameter überschrieben werden.

13. FAQ und Tipps

Wie verbinde ich mich mit dem IBKR Gateway?

Verwenden Sie den connect-Befehl. Für die Standard-Docker-Konfiguration reicht connect default. Für manuelle Verbindungen geben Sie Host, Port und Client-ID an.

Wie funktioniert die Auto-Discovery (Connection 3.0)?

Der PTA versucht automatisch:

  1. Verbindung zu localhost:4002 (Paper Default)
  2. Fallback zu localhost:4001 (Live)
  3. Bei Misserfolg: Nachfrage beim User nach Custom IP/Port

Wie erstelle ich eine Market Order?

Geben Sie keine limit_price an:

trade {"action": "ENTER", "ticker": "NVDA", "quantity": 10}

Wie erstelle ich eine Limit Order?

Geben Sie limit_price an:

trade {"action": "ENTER", "ticker": "NVDA", "quantity": 10, "limit_price": 850.00}

Wie finde ich die trade_id einer Position?

Verwenden Sie trades um alle offenen Positionen mit ihren IDs aufzulisten, oder status für einen vollständigen Portfolio-Snapshot.

Was bedeutet der Heat-Index?

Der Heat-Index ist das Gesamtrisiko aller Positionen als Prozentsatz des Kontowerts. Ein Heat-Index über 2.5% pro Position löst eine Warnung aus. Das Gesamtrisiko sollte typischerweise nicht 10% des Kontowerts überschreiten.

Wie funktioniert die Minervini-Positionsgrößen-Berechnung?

Der wizard-Befehl berechnet die optimale Positionsgröße basierend auf:

  1. Risiko pro Trade (typisch 1% des Kontos)
  2. Maximale Positionsgröße (typisch 25% des Kontos)
  3. Verfügbarem Cash

Die kleinste der drei Berechnungen wird als empfohlene Anzahl ausgewählt.

Wie rufe ich historische Performance ab?

Verwenden Sie analyze history {"days": 30} für die letzten 30 Tage. Für ticker-spezifische Analysen: analyze history {"days": 90, "ticker": "NVDA"}.

Welche Metriken werden in der Trade-Analytik berechnet?

  • Winrate: Prozentsatz der gewinnbringenden Trades
  • Avg Win / Avg Loss: Durchschnittlicher Gewinn/Verlust
  • Payoff Ratio: Verhältnis von avg_win zu avg_loss
  • Profit Factor: Gross Profit / Gross Loss
  • Expectancy: Erwarteter Wert pro Trade
  • SQN (System Quality Number): Qualitätsmetrik für das Handelssystem

Wie funktioniert der PTA?

Der PTA verwendet einen LLM (Local Language Model) über LM Studio. Er versteht natürliche Sprache und führt automatisch CLI-Befehle aus. Der PTA:

  1. Versteht die Benutzeranfrage
  2. Generiert den entsprechenden CLI-Befehl als Tool-Aufruf
  3. Führt den Befehl über den CLI-Controller aus
  4. Verarbeitet das Ergebnis und formuliert eine Antwort

Tipps für den PTA

  • Verwenden Sie klare, spezifische Anweisungen
  • Der PTA prüft immer zuerst IDs mit status oder trades vor Exits/Updates
  • Shortcut-Befehle: "risk" führt analyze live aus, "Performance" zeigt die letzten 30 Tage

Wo werden Daten gespeichert?

  • data/trades//: Trade-JSON-Dateien
  • data/portfolio_latest.json: Letzter Live-Snapshot
  • logs/communication.log: PTA-Kommunikationsprotokoll

Wie führe ich die Tests aus?

Verwenden Sie pytest:

pytest tests/                    # Alle Tests
pytest tests/ -v                 # Detaillierte Ausgabe
pytest tests/test_cli_analyze.py # Einzelner Test
